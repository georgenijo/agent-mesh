# Audit: what agent-mesh can take from multi-agent-pm

> Source: https://github.com/shawngeorgie06/multi-agent-pm (backend, ~11k LOC).
> Method: 8 parallel subagent auditors (one per subsystem cluster) → 1 synthesis pass.
> Date: 2026-06-04.

## 1. Verdict

There is genuinely reusable material here, but only at the level of *patterns*, never code — every mechanism in multi-agent-pm assumes one Node event loop and plain in-memory Maps, which is the exact opposite of agent-mesh's separate-process/container reality. The highest-value steal is the **atomic guarded-claim primitive** (`updateMany where claimedBy:null`, count==0 == lost): it's the one correctly-designed concurrency primitive in the codebase and ports cleanly to a JetStream KV revision-CAS. The single biggest *avoid* lesson is structural and repeats in every cluster: **multi-agent-pm constantly maintains two unreconciled sources of truth** (Prisma claim vs in-memory FSM, two queues, two heartbeat timers, RAM history vs DB) — agent-mesh must have exactly ONE authority per fact (the JetStream record) and put the validating FSM in the write path, not beside it. Treat the project's own "production-ready / 379 tests" docs as aspiration: the load-bearing orchestration layer (Phase 5) has zero non-test importers and the live server runs with persistence silently disabled.

## 2. Steal — ranked

1. **Atomic guarded-claim as the single-winner lock** — A conditional write whose affected-row count *is* the lock result: `updateMany({where:{taskId, claimedBy:null}, data:{claimedBy}})`, then `count===0` means "someone else won". No read-then-write window, no separate lock service. *(AgentClaimingHelper.ts:45-64; also TaskQueueManager.ts:338-369.)* **Adapt:** This is the real mechanism behind both `announce` pre-edit locks and ask-ticket claim. Implement as a create-only / revision-guarded `put` on a JetStream KV key (`mesh.claim.<path>` or `mesh.ticket.<id>`); the failed revision check is the "already claimed" signal. Return a **typed** result `claimed | lost | error` so a sidecar retries on transport error but not on a legitimate loss.

2. **Async respond-then-work / ask-returns-a-ticket** — Acknowledge cheaply, do the expensive LLM work off the response path. `POST /api/projects` validates, persists, `res.json({projectId})` immediately, then runs the workflow asynchronously over the socket. *(server.ts:262-375; mirrored in AgentOrchestrator.ts:468-543.)* **Adapt:** This *is* the async-by-default law. `mesh ask` persists a ticket in JetStream and returns a ticket id synchronously; the responder's turn happens out-of-band and surfaces via `poll`/`inbox`. **But** the project's `request()` (MessageBus.ts:225-274) does the opposite — awaits the peer inline — so use *that* as the anti-pattern reference, and never await a Promise for an answer.

3. **Capability-tagged agent card declared at join** — `{agentId, agentType, capabilities[]}` declared once in the constructor and carried as a unit; coarse closed-enum role for pub/sub addressing + open string-array caps for fine predicate matching. *(BaseAgent.ts:23-27,46-60; caps at FrontendAgent.ts:182, BackendAgent.ts:57, etc.)* **Adapt:** This is the mesh agent-card emitted on `join` and surfaced by `who`. `agentType → --role`, `capabilities[] → --caps`. Make role/caps **open data registered at join**, not a closed compile-time union (their enum is duplicated in two files — BaseAgent.ts:25 and AgentClaimingHelper.ts:12 — and a third-party CLI like Cursor/Aider *cannot* register a role it ships). Treat advertised caps as claims to verify (handshake/probe), not ground truth.

4. **Pull-based routing: control-plane suggests, agent commits** — Routing emits to a type-scoped channel (`agent:${agentType}`) and lets any capable peer self-claim via CAS; the coordinator picks a "best" agent only for load accounting but still broadcasts to the whole type, so stale coordinator state can't strand work. *(TaskDistributionService.ts:451-460; AgentTeamOrchestrator's control-plane is pure registry+roles, never touches payloads — AgentTeamOrchestrator.ts:234-295.)* **Adapt:** Coordinator publishes a ticket to a capability/role subject (`mesh.role.frontend`); sidecars subscribe and the first to win the KV CAS claims it. Keeps the coordinator off the data path — it computes the subject, never the assignment.

5. **Two-tier heartbeat lease (liveness vs GC) + reclaim on death** — Separate "temporarily unreachable → OFFLINE" from "gone → auto-unregister at 2× timeout", with a registration grace period so a fresh agent isn't reaped before its first beat; on eviction, re-fail the dead agent's in-flight task back to the queue. *(ParallelExecutionEngine.ts:842-893, 251-259; recovered-event + missed-beat debounce at ExecutionMonitoringService.ts:211-233,288-310.)* **Adapt:** Sidecar heartbeats over NATS; coordinator runs the two-tier logic to mark an agent `away` (still in `who`/`status`, degraded) vs evict. **Critical:** on eviction, re-open any ask-tickets that agent had accepted-but-not-answered — otherwise crashed responders leave tickets pending forever. This also fixes the project's own gap: a hard-crashed agent never emits `agent:offline`, so without a TTL lease its claim is held forever (AgentClaimingHelper.ts:45-94).

6. **Declarative FSM as data, in the write path** — Legal transitions as one `Map<State,Set<State>>` table (terminal = empty set), with `isValidTransition`/`getValidNextStates` derived from it, plus a per-entity append-only transition history. *(TaskStateMachine.ts:23-30,181-200,139-153.)* **Adapt:** Encode the ask-ticket lifecycle (`open → accepted → answered → closed/expired`) as one table the sidecar/coordinator validates **before** publishing, so a peer physically can't answer an already-expired or already-answered ticket. Make the JetStream event stream (`mesh.ticket.<id>.events`) the history — no in-memory map. Their FSM was correct but **dead code**, never called by the claim path (see Avoid).

7. **Single transport facade owning the wire format** — One method per semantic event, callers never touch the socket directly, invariants enforced centrally (progress clamped `Math.min(100,Math.max(0,p))` at the publish edge). *(SocketServer.ts:238-388; clamp at :268.)* **Adapt:** The sidecar is the single chokepoint that owns the NATS subject + envelope catalog. Agents call high-level verbs (`announce`/`note`/`answer`); the sidecar shapes, validates, clamps, and publishes. Keep one **encode+decode** module (the project co-locates them — MessageParser.ts:297-312 format vs parse) so heterogeneous CLIs can't each invent a format — but make it versioned JSON/CBOR (`schemaVersion`+`kind`), not markdown.

8. **Self-describing envelope + correlation/threading keys** — Routing metadata split from opaque payload: `{id, from, to, channel, content, ts, threadId?, parentMessageId?, requestId?}`; `threadId` + `parentMessageId` give a cheap conversation graph; namespaced subject-like event names (`execution:task:claimed`). *(MessageBus.ts:32-42, 466-482; SocketServer.ts namespaced names.)* **Adapt:** Maps to the NATS envelope for ask/answer. Keep `requestId`/`threadId` as first-class correlation keys (threadId = ticket/conversation id, parentMessageId = clarification chain), but store the thread index in JetStream KV, not an in-memory Set, so it survives restart and is cross-process visible. Make `channel` a NATS subject (`mesh.ask.<role>`).

9. **Dependency-gated readiness, completion-as-clock** — A task is offered only when all deps are COMPLETE; any completion re-triggers evaluation of the unblocked set; DFS cycle-check at insert time. *(TaskDistributionService.ts:321-349; TaskQueueManager.ts:632-702.)* **Adapt:** Event-driven DAG scheduling with no separate scheduler — completion events are the clock. Model "ask X depends on note Y" on the JetStream blackboard; route the ask only once prerequisite notes exist; reuse the cycle-check before accepting a dependency edge. (Scope re-evaluation to the affected DAG — see Avoid #5.)

10. **Degrade-don't-throw parsing at the boundary** — `parseMessage` wraps all extraction in try/catch and returns a minimal valid result on failure; per-role typed output contracts with tolerant parsers + deterministic fallbacks. *(MessageParser.ts:7-41; ResearchAgent.ts:185-225, QAAgent.ts:221-254.)* **Adapt:** A sidecar decoding an answer from a foreign CLI must never crash the responder's turn on a malformed envelope — return a typed `unparseable` result and surface it. Adopt the discipline, but enforce a strict structured envelope so this is a rare error path, not the normal decode path (their regex decoding is itself an anti-pattern — see Avoid #6).

## 3. Avoid — ranked

1. **Two unreconciled sources of truth for the same fact** — Pervasive. Postgres claim (raw status strings) vs in-memory `TaskStateMachine` that the claim path never calls and that uses a *different vocabulary* (DB has no CLAIMED/RETRYING; FSM has no BLOCKED) *(AgentClaimingHelper.ts:69-76 vs TaskStateMachine.ts:23-30)*; two entirely separate queues with different agentType semantics *(TaskDistributionService Prisma queue vs TaskQueueManager in-memory — :297-306 vs :141,:180-229)*; two heartbeat timers with conflicting thresholds (60s vs 30s) *(ParallelExecutionEngine.ts:158 vs ExecutionMonitoringService.ts:79-83)*; two phase state machines that don't know about each other *(AgentTeamOrchestrator.ts:480-646 vs PhaseOrchestrator.ts:63-204)*. **Mesh rule:** ONE authority per fact (the JetStream record); every other consumer subscribes to its verdict and never re-derives it.

2. **Fake atomicity** — `TaskQueueManager.claimTask` literally comments "Simulate atomic operation" and is a Map read-recheck-write that's safe only because Node is single-threaded; its `version` field is hardcoded to `1` and never used for CAS *(TaskQueueManager.ts:338-356,368)*. Even the real Prisma path throws away its atomicity with unwrapped follow-up writes (claim TaskQueue, then separately update Task, then broadcast — no transaction, so a crash between them = split-brain) *(AgentClaimingHelper.ts:48-89)*. **Mesh:** the claim commit must be a single atomic KV op; derived state is projected *from* that one record, never written as a second step. Two separate sidecars will both read-unclaimed and both claim under this pattern.

3. **No lease/TTL; release only via graceful event** — `claimedAt` is written but never read to expire stale claims; the only release path is a cooperative `agent:offline` broadcast *(AgentClaimingHelper.ts:45-94; TaskDistributionService.ts:215-249)*. A hard-crashed/OOM-killed container never emits offline → claim held forever, task stranded. Same shape in transport: presence is edge-triggered online/offline events, so a crashed agent shows forever-online *(SocketServer.ts emitAgentOnline/Offline)*. **Mesh:** every claim and presence record is a TTL lease renewed by heartbeat.

4. **Timeout resolves as fake success; no real error channel** — `request()` resolves with a synthetic `{error:'timeout'}` instead of rejecting, and `reject` is a permanent no-op `() => {}` *(MessageBus.ts:228-244)*. Callers can't distinguish a real answer from a timeout except by string-matching a magic field. Same disease in claim error handling: `claimTask` returns `false` both when another agent won and when Prisma threw *(AgentClaimingHelper.ts:60-64 vs 90-93)*. **Mesh:** every verb returns an explicit typed state (`answered | timed_out | error`, `claimed | lost | error`); never a fake-success payload.

5. **Fan-one-event-into-a-full-rescan + N+1** — Routing re-triggers globally on nearly every event (`agent:online` scans ALL projects), each doing multiple Prisma queries plus an N+1 (`findUnique` + `findMany` per task) inside the batch loop *(TaskDistributionService.ts:138,196-209,316-319,410-425)*. O(events × tasks) DB amplification, thundering-herd on every completion. **Mesh:** scope re-evaluation to the affected DAG/role and debounce; never fan one completion into a full rescan.

6. **Decoding the agent contract from markdown with stacked regexes** — Message type, tasks, effort, priority, dependencies, questions, blockers all scraped by regex; any line ending in `?` is treated as a question *(MessageParser.ts:46-292, esp. 224-232)*. For heterogeneous CLIs this *is* the cross-CLI parity trap — different agents phrase differently and silently misparse. **Mesh:** mandate a structured JSON envelope (`schemaVersion` + typed fields) on the wire; treat free-text only as opaque `content`.

7. **Substring / bidirectional capability matching + inferred roles** — `cap.includes(req) || req.includes(cap)` matches `'js'` to `'typescript-jsx'`, `'api'` to `'restapi'`, `'go'` to `'mongo'` *(AgentClaimingHelper.ts:34-38)*; routing identity reverse-engineered from substrings of taskId/description with a hardcoded `'LAYOUT'` fallthrough *(TaskDistributionService.ts:598-641; PhaseOrchestrator.ts:314-350)*; two layers even disagree (substring vs exact-set), so the same task routes differently depending which runs. **Mesh:** exact/structured capability tokens with set-membership; never infer roles from free text; never collapse unknowns to a default worker.

8. **Unbounded in-memory growth, no eviction** — `directMessages`, `messageHistory`, `threads`, `pendingRequests`, `taskExecutionHistory`, `deadLetterQueue`, per-agent `conversationHistory`, the `agentAvailability` registry — all append-only, cleared only by a test-only `reset()` *(MessageBus.ts:162-371; ParallelExecutionEngine.ts:138-167; ResearchAgent.ts:116-129; TaskDistributionService.ts:216-222)*. **Mesh:** JetStream retention/TTL/compaction from day one; bounded inbox depth with backpressure, not RAM buffering; aggregate queries must not scan all-time history per call.

9. **Silent no-op persistence / fire-and-forget saves** — The live server constructs the bus and queue with **no store**, so all "durable" persistence quietly does nothing and is lost on restart, zero error raised *(server.ts:22, :303; guarded `if(!taskStore)return` at MessageBus.ts:386,405,488)*; writes that do happen use `.catch(console.error)` with RAM as a second unreconciled source *(MessageBus.ts:212-215)*. **Mesh:** durability is write-then-ack against JetStream; fail loudly if the store binding is missing rather than degrading to demo-only in-memory mode.

10. **Weak IDs, fire-after-teardown timers, leaked listeners, no scoping** — `Date.now()+Math.random().substr(2,9)` ids collide and aren't sortable across processes *(MessageBus.ts:226,328,474)* → use ULID/UUIDv7. `setTimeout` retries with no handle fire into torn-down state *(ParallelExecutionEngine.ts:593-600)*. `setupEventListeners` adds `.on()` handlers `stopListening` never removes *(BaseAgent.ts:125-147)*. `getTeamStatus()` ignores projectId and returns the last team inserted via `.pop()` *(AgentTeamOrchestrator.ts:358-409)*. **Mesh:** time-ordered ids; every scheduled action cancellable and tied to entity lifecycle; tear down subscriptions on leave/reconnect; every control verb explicitly scoped by mesh/agent id.

11. **Global firehose with no auth or scoping** — Every emit is `io.emit` to ALL clients; rooms helpers exist but are never wired; CORS `'*'` and zero handshake auth, so the full multi-project agent feed is effectively public *(SocketServer.ts:45-47,7,393-395; server.ts:47)*. **Mesh:** the dashboard taps `mesh.>` via real subject filters and an authenticated subscriber, not a global broadcast filtered client-side.

12. **Poison-task infinite retry** — `reportFailure`/`task:failed` resets straight to TODO with no attempt counter or cap, despite `maxRetries`/`RETRYING` existing *(AgentClaimingHelper.ts:134-159; TaskDistributionService.ts:156-170)*. **Mesh:** tickets/claims carry an attempt count and a terminal give-up state (dead-letter inbox of unanswered/abandoned tickets — the one genuinely good idea at ParallelExecutionEngine.ts:544-645).

## 4. Cross-cutting observations

- **Single-event-loop is load-bearing and invisible.** Every "atomic" claim, every pub/sub "delivery", every FSM is a synchronous in-heap Map mutation. The code looks concurrency-safe only because Node never interleaves. This is the *one* assumption agent-mesh categorically cannot inherit — separate processes/containers remove the very property that makes their code correct. Mining is pattern-only; zero transport reuse.
- **Coupling of control-plane and data-plane in handlers.** The whole three-phase pipeline lives inside the `POST /api/projects` body (server.ts:262-375) — untestable without the full agent stack, non-resumable. *Validates* the mesh law (the project also has a clean pure-registry control-plane in AgentTeamOrchestrator that touches zero payloads — proof the split is feasible) and warns against fusing accept-request with run-work.
- **Infrastructure built ahead of need, then half-wired.** Phase 5 orchestrator has zero non-test importers; `TaskStateMachine`, `TaskQueueManager.findAgentForTask`, dead metrics (`distributionSuccessRate` init 100 never updated; `phaseCompletionTimes` read but never written; `metricsCache` written but never served) — features structurally unreachable because the producing write was never connected to the consuming read. **Lesson:** gate "done" on end-to-end tests across real separate processes, not green unit tests over mock stores.
- **Docs overstate maturity.** "Production-ready / 379 tests / 84% coverage" vs a server that runs persistence-disabled and never instantiates the orchestration layer; completion reports are LLM-authored and self-congratulatory, contradict each other on provider/agent-count/port/purpose, and reference a different machine's paths. Read them as design intent, never as validated behavior.

## 5. Concrete next actions for agent-mesh

- **Pick JetStream KV revision-CAS as the single claim/lock primitive** for both `announce` pre-edit locks and ask-ticket claim; return typed `claimed | lost | error`. Decide now: advisory `announce` (pub/sub) *plus* a real CAS lock for actual edits, not advisory-only.
- **Make every claim and presence record a TTL lease** renewed by sidecar heartbeat; on lease expiry, the coordinator marks the agent away and **re-opens that agent's accepted-but-unanswered tickets**.
- **Define ONE versioned envelope module** (encode+decode co-located, `schemaVersion`+`kind`, JSON/CBOR) owned by the sidecar; one NATS subject taxonomy (`mesh.<verb>.<role>` / `mesh.ticket.<id>.events`); ULID/UUIDv7 ids. Forbid any CLI from touching the bus directly.
- **Encode the ask-ticket FSM as a data table validated in the write path** (sidecar/coordinator rejects illegal transitions before publish); the JetStream event stream *is* the history.
- **Lock the async contract:** `ask` persists a ticket and returns an id synchronously; answers arrive via `poll`/`inbox`; no verb ever awaits a peer's reply. Add a dead-letter inbox for unanswered tickets with bounded attempts.
- **Make roles/caps open data registered at join**, exact-token matched, with a capability-verification handshake (probe) — not a closed enum, not substring matching, not self-asserted ground truth.
- **Keep the coordinator a pure reducer over join/leave/status/heartbeat** — registry, role→subject, policy only. It computes subjects and presence; it never sits on ask/answer.
- **Fail loudly on a missing JetStream binding**; set retention/TTL/compaction and bounded inbox depth from day one; derive presence/health aggregates by counting the registry at read time, not hand-rolled +/- counters.
- **Authenticate the dashboard subscriber** and scope it via subject filters; treat the read-only `mesh.>` tap as the model (validated by their read-only socket), but never the unscoped, unauthenticated firehose.
