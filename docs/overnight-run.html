02:58 · LAUNCH

Coordinator, dashboard, review gating, auto-experts — all green. The control plane never wobbled all night.

RUN 1 · COLLAPSE

Reviews ran on the wrong model bug

MESH_EXPERT_MODEL was a silent no-op — reviewers ran the account default, not opus. And one request_changes on a foundational task cascade-cancelled whole jobs.

~03:30 · HAND-FIX

Shipped the expert-model fix + #85 fixed

Plumbed the expert model through; built #85 so request_changes re-dispatches with feedback instead of sinking the job. Tested, pushed.

RUN 2 · COLLAPSE

One reviewer, eight workers bottleneck

A single opus reviewer serialized every review at 45–110s each. With 8 workers the queue blew past the 5-minute gate — 15 reviews timed out → cascade. Throughput ceiling, not bad code.

~04:28 · RE-TUNE

Throughput-matched the fleet tuned

Capped workers to the reviewer's rate, widened the gate to 20m, and added cost logging so spend survives an unlimited budget.

RUN 3a · COLLAPSE

Every issue-job died at spawn bug

--issue owner/repo#N stored the full slug as the repo; the worker rejects the slash as a path-traversal guard. Workers never started. Worked around with --repo agent-mesh.

RUN 3b · THE DEEP ONE

The fleet was strangling itself critical

Worker teardown never killed its claude child. 29 zombie sessions piled up and exhausted the account's concurrency, so new workers hung and timed out — spawning more zombies. A perfect loop.

05:21 · CLEANUP + FINAL TEST

Cleared the zombies, serialized to one isolated

Surgically killed only the fleet zombies (left the human's own sessions alone), dropped to a single worker. The cleanest possible shot at one green job.

05:48 · VERDICT

Still no clean end-to-end job 0 merges

Even serialized, the worker turn couldn't complete. Stopped tuning. The night's value isn't merged code — it's the bug list below.

What dogfooding surfaced

CRITICAL1 · Worker leaks its claude child

Teardown doesn't kill the runtime process on timeout. Zombies accumulate until the account's concurrency is exhausted and long runs strangle themselves. Fix first — nothing else matters until this is closed.

CRITICAL · ARCH2 · One expert is a fleet-wide throughput ceiling

A single reviewer serializes every review and every ask. The highest-leverage change is a reviewer/expert pool, or binding worker concurrency to review throughput.

HIGH3 · Issue ingestion stores the wrong repo

job.repo keeps the full owner/repo slug; the worker rejects the slash. Normalize to the last path segment.

HIGH4 · Review gate + cascade-cancel is brittle

#85 softens it, but one reject still sinks a whole multi-task job. Consider per-task isolation.

MEDIUM5 · Planner over-decomposes

10–13 tasks for modest features multiplies load on the single expert. Give decomposition a budget.

LOW · FIXED6 · Cost unobservable at unlimited budget

Now logged per accrual. (Values are $0 — this is a subscription account, not a missing metric.)
